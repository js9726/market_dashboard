# Market Dashboard

Future multi-client trading journal and market intelligence SaaS.

This app is evolving from Jie Sheng's personal trading workstation into a SaaS product for market dashboards, trade journaling, AI trade reviews, scanners, watchlists, and eventually broker-powered workflows.

GitHub: `js9726/market_dashboard`

---

## Current Product Shape

| Area | What it does today |
|---|---|
| Market dashboard | Static market snapshot generated by Python and served by Next.js |
| Morning brief | Multi-provider AI brief generated from market data and web search |
| Journal | Google Sheets connection, trade sync, stats, calendar, trade log |
| Trade review | LLM verdicts using runtime prompt skills and trader personas |

| Auth | NextAuth v5 with Google OAuth, Prisma, and user roles |
| Deployment | Vercel for the Next.js app, GitHub Actions for scheduled data refresh |

## Strategic Direction

This repository should be treated as a **future multi-client SaaS**, not only a personal tool.

That means:

- Every user owns their own journal connection, trades, watchlist, scan results, quotas, and verdict history.
- Broker or data-provider integrations must be designed as tenant-aware services.
- Personal local files, Google Sheet IDs, and broker details must not be hardcoded into shared runtime code.
- Runtime prompt/knowledge modules must be committed with the app so Vercel can deploy them.
- Local/global assistant skills remain the authoring and operator layer, but production cannot depend on `C:\Users\...` paths.

## Website Vs Skills

There are three separate layers that should not be mixed up:

| Layer | Location | Purpose |
|---|---|---|
| Authoring wiki | `../llm_traders_wiki/wiki/` | Source knowledge: trader styles, risk rules, setups, company dossiers |
| Global assistant skills | `C:\Users\jiesh\.claude\skills` and `C:\Users\jiesh\.codex\skills` | Personal AI workflows: trade analysis, MooMoo API helper, B&Q repricer |
| SaaS runtime skills | `packages/core-skills/` | Deployable prompt/knowledge/handler modules used by this app |

The long-term rule:

1. Edit knowledge in `llm_traders_wiki` or global skills first.
2. Sync or generate runtime artifacts into `packages/core-skills/` using the formal manifest.
3. The website calls `packages/core-skills/` in production.
4. Personal assistant sessions may read the live local/global skills directly.

This gives one authoring source while keeping the SaaS deployable.

## MooMoo API Direction

MooMoo/OpenD is **not wired into this app yet**.

Current app market data uses:

- Python `yfinance`
- Next.js `yahoo-finance2`
- Finviz scraping
- Google Finance fallback in the Python pipeline

Future MooMoo direction:

- Keep the global `moomooapi` and `install-moomoo-opend` skills as the operator/source-of-truth workflow.
- Add a deliberate broker data service when the SaaS is ready.
- Start with the owner's MooMoo connection only, then require each future user to connect their own MooMoo/OpenD for quotes, paper trading, backtesting, and any broker-powered features.
- Use MooMoo for ticker quotations and broker-owned market data when a user has connected it; keep Yahoo/yfinance style feeds as public fallback data.
- Start with read-only/paper-trading features: quotes, account snapshot, positions, order status.
- Require explicit confirmation for any live trading action.
- Do not assume Vercel can talk directly to a local OpenD process. A production design needs a secure bridge, hosted worker, or user-side connector.

See `docs/moomoo-saas-broker-service.md` for the service boundary.

## Runtime Skills

Runtime skills live in `packages/core-skills/`.

They are not the same thing as Claude/Codex global assistant skills. They are deployable app modules with:

```text
SKILL.md
prompt.md
knowledge.md
schema.json
handler.ts
handler.py
tests/golden.json
```

Current runtime skills include:

- `morning-brief`
- `trader-scorer-market`
- `trader-scorer-stock`
- `trader-scorer-trade`
- `trader-primetrading`
- `agent-moderator`

When trading knowledge changes, update `llm_traders_wiki` first, then refresh the relevant runtime skill.

Skill sync is controlled by:

```text
packages/core-skills/skill-sync.manifest.json
```

Run from `apps/market_dashboard/`:

```bash
npm run skills:check
npm run skills:sync
npm run skills:sync -- --skill trader-primetrading
```

`skills:check` warns when upstream wiki/global skill content has changed after a runtime skill has been synced. Vercel and any hosting provider receive only the committed `packages/core-skills/` files; they do not read your local wiki or global Claude skill folders.

## Auth Model

The app uses NextAuth v5 with Google OAuth and Prisma.

Roles:

- `owner`: admin
- `allowed`: approved user
- `pending`: signed in but waiting for approval
- `denied`: blocked

Every `/api/*` route should include its own `auth()` check and return JSON `401` for unauthenticated API calls. Middleware redirects are not enough for API safety.

## Data Safety

- Always write JSON through the safe JSON helpers in the Python pipeline.
- Bare `NaN` breaks browser `JSON.parse`.
- Do not commit `.env`, `.env.local`, API keys, broker credentials, or personal account data.
- Treat generated `data/` and `public/market-dashboard/` files as artifacts.
- Keep personal broker workflows separate until a SaaS-grade broker integration is designed.
